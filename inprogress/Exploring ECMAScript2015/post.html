Arrow function
The "fat" arrow => (the opposed to the thin arrow ->) was chosen to compatible with CoffeeScript style. Specifying parameters: [javascript] () => { ... } // no parameter x => { ... } // a one parameter (x, y) => { ... } // several params [/javascript] Specifying body: [javascript] x => {return x * x} // block style x => x * x //expression, equivalent to previous line let squares = [1, 2, 3].map(x => x * x) // a real example [/javascript] There are two benefits to use arrow functions:
  1. They are less verbose than traditional function expression
  2. Their this picked up from surrounding (lexical scope), e.g. you don't need to use bind or self = this
Traditional functions can be used as:
  1. Non-method function
  2. Methods
  3. Constructors
The problems appear in 2 and 3, when function has always their own this. Example of problem in ES5: [javascript] function Prefixer(prefix) { this.prefix = prefix; } Prefixer.prototype.prefixArray = function (arr) { 'use strict'; return arr.map(function (x) { // Doesnâ€™t work: return this.prefix + x; // this = undefined, in "non-strict" mode this = window }); }; [/javascript] Solution with self = this: [javascript] function Prefixer(prefix) { this.prefix = prefix; } Prefixer.prototype.prefixArray = function (arr) { var self = this; // store in closure return arr.map(function (x) { return self.prefix + x; }); }; [/javascript] Solution with specifying value for this: [javascript] function Prefixer(prefix) { this.prefix = prefix; } Prefixer.prototype.prefixArray = function (arr) { return arr.map(function (x) { return this.prefix + x; }, this); // a few array methods have an extra last parameter for specifying value to "this" }; [/javascript] Solution with bind(this): [javascript] function Prefixer(prefix) { this.prefix = prefix; } Prefixer.prototype.prefixArray = function (arr) { return arr.map(function (x) { return this.prefix + x; }.bind(this)); // now this not depends of context calling, "this" is always the same value }; [/javascript] ES2015 solution: [javascript] function Prefixer(prefix) { this.prefix = prefix; } Prefixer.prototype.prefixArray = function (arr) { return arr.map((x) => { return this.prefix + x; }); }; //fully ES2015-ify the code: class Prefixer { constructor(prefix) { this.prefix = prefix; } prefixArray(arr) { return arr.map(x => this.prefix + x); // has lexical "this", its determined by the surrounding scope } } [/javascript] IIFE as arrow functions: [javascript] (() => { return 123; })(); [/javascript]
Classes
Classes are syntactical sugar over existing prototyped-based inheritance. The class syntax doesn't introduce new object-orient inheritance model to JS. It just provides a much simpler and cleaner way to create object and dealing with inheritance. this.color = color; } toString() { return super.toString() + ' in ' + this.color; } } let cp = new ColorPoint(25, 8, 'green'); cp.toString(); //'(25, 8) in green' cp instanceof ColorPoint; //true cp instanceof Point;
